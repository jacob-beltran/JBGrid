Introduction

Hello there! And welcome to the example/documentation page for the JBGrid Responsive CSS Grid System. It is built for modern browsers, which means all recent versions of Chrome, Safari, and Firefox are supported. Internet Explorer is supported from version 9 and up. There are many responsive grid systems available on the Internet. Many of them are more mature and well tested than this one. You should probably use one of those. This measly thing is the result of some guy just wanting to experiment with SASS and who simply tried to make something moderately useful out of it. There is really no reason to use this other than morbid curiosity, or possibly you just like making your life difficult by using untested tools that are unlikely to be maintained.

Oh you're still here. Well great! I, for one, am glad you're still interested in this contraption. Okay, so a little background information on this grid system. JBGrid is a twelve column responsive grid system. It has 5 breakpoints defined to allow for a decent amount of flexibility without making things too terribly complicated. FYI the breakpoints are defined in em units. This is done so that zooming in the browser doesn't break the responsiveness of everything. The breakpoints are defined as follows:

  • Extra Small"xs":0em - 20em
  • Small"s":20em - 35em
  • Medium"m":35em - 60em
  • Large"l":60em - 80em
  • Extra Large"xl":Larger than 80em

Now that we have a little background information, let's dive into the grid system itself to see how it works.

Grid Basics

In a nutshell, you can size your elements for each viewport by applying a class for each breakpoint you want to use along with the width that the component should take at that breakpoint. The class names used are a combination of one of the sizes defined above and the number of columns wide that the column should be. Examples of some column class names would be things like "xs-12" (where the element would span 12 columns on extra small screens an up), "m-6" (where the element would span 6 columns on a medium sized screen and up, and "l-4" (where the element would span 4 columns on a large screen. Note that any element designated as a column must also have the "column" classs applied to it in order for it to receive the appropriate width.

Responsive Design

Responsive Web Design is very easy with JBGrid. Simply set classes for each breakpoint you want to use. Breakpoints are calculated using the min-width CSS media query function, so unless another class is specified the lowest sized breakpoint rules are always applied. So a column div with only the class "xs-12" will span 12 columns for all viewport sizes unless another breakpoint size is defined, such as "l-6". In that case, all sizes from Extra Small to Medium will still fall under the Extra Small breakpoint rule, but viewports that are of the size Large and up will be defined using that breakpoint and will span 6 columns.

You can get as granular as you would like when specifying breakpoints. You can use all 5 sizes from "xs" to "xl". Or you can just specify one width for all viewports. Note that when specifying widths, it is best practice to always include a specification for the smallest viewport size, "xs". Due to the nature of the responsive grid system, the "xs" defined widths will always be used if no other widths are specified, but even if there are other widths specified, the Extra Small width should always be set. Note: generally, it is best practice to set a width of 12 columns for Extra Small sized viewports, to maximize the area for your content on the smallest of screen sizes.

One word on the sizes defined in the Introduction section. It may be helpful to know which sizes correspont to which size of device. That information is included below:

  • Extra Small"xs":Very Small Phones and Smartwatches.
  • Small"s":Most Phones in Portrait.
  • Medium"m":Most Tablets. Large Phones in Landscape.
  • Large"l":Most Laptops. Large Tablets in Landscape.
  • Extra Large"xl":Most Large Desktop screens.

Fancy Stuff

There are a few "fancy" features built into the JBGrid system. Some of these features require the JavaScript helper for JBGrid. This helper can be found in the "js" folder with the filename jbgrid.js or jbgrid.min.js. The two files are functionally identical. The minified file should be used in production since it is of a smaller file size.

Padding Columns

The JBGrid css file has classes buit into it for adding amounts of padding to columns. Using the padding functionality does not require the JavaScript helper. The padding classes are as follows:

  • "padding-xs"
  • "padding-s"
  • "padding-m"
  • "padding-l"
  • "padding-xl"

Though they may look similiar to the breakpoint classes used for columns, their behavior is different. Notice that the size is not the first element of the class name. The above classes are used to add differeing amounts of padding to elements. The "padding-xs" class will add an "Extra Small" amount of padding to the element. The "padding-m" class will add a "Medium" amount of padding to the element. The "padding-xl" class will add an "Extra Large" amount of padding to the element.

The exact amount of padding added for each one of these classes depends on the size of the viewport. For the Extra Small padding class, the amount varies from 2px (on very small devices) to 6px (on very large screens). For the Extra Large padding class, the amount varies from 10px (on very small devices) to 14px (on very large screens). The amounts for the other classes lie somewhere in the middle. The point is, the amount of padding will grow larger depending on the viewport when using these classes.

Matching Column Height

In some cases, you may want neigboring columns to match each other's height. This can be difficult if we don't know the exact height of the largest element. JBGrid provides classes for matching the height of neighboring elements. Note that this functionality requires the JavaScript helper. The clases that can be used for this functionality are listed below. These classes must be applied to the "row" element that contains the columns to be matched.

  • "fill-height":Will match height in all viewports.
  • "fill-height-xs":Will match height in Extra Small viewport only.
  • "fill-height-s":Will match height in Small viewport only.
  • "fill-height-m":Will match height in Medium viewport only.
  • "fill-height-l":Will match height in Large viewport only.
  • "fill-height-xl":Will match height in Extra Large viewport only.

Setting Aspect Ratios

One nifty feature that exists in JBGrid is the ability to define aspect ratios for grid elements. This is done by using data- attributes. The following data-attributes will set aspect an aspect ratio when they are defined for an element.

  • "data-aspectratio":Defines an aspect ratio for all viewport sizes.
  • "data-aspectratio-xs":Defines an aspect ratio for Extra Small viewport sizes.
  • "data-aspectratio-s":Defines an aspect ratio for Small viewport sizes.
  • "data-aspectratio-m":Defines an aspect ratio for Medium viewport sizes.
  • "data-aspectratio-l":Defines an aspect ratio for Large viewport sizes.
  • "data-aspectratio-xl":Defines an aspect ratio for Extra Large viewport sizes.

As you can see, aspect ratios can be defined per viewport size. So components can have different aspect ratios on different screen sizes. The value for each of these attributes is an aspect ratio in the format {width}x{height}. For example, 16x9, 4x3, and 2x1 are all valid aspect ratios. The width and height components must be integers. There is also a shorthand value square for creating square components.

Using "Sticky" Positioning

As you can see (if you are on a large enough screen), the link menu to the left remains fixed after scrolling past a certain point. This is called "sticky" positioning. There is preliminary support in some browsers for doing this in pure CSS, but browser support isn't quite there yet. In addition, if you want to change the position of an elment to fixed, it removes that element from the grid context and content is unpredictably shifted around. The sticky positioning available in JBGrid, preserves the grid context, so you do not have to worry about your layouts changing when the position is shifted.

This "sticky" positioning can be accomplished by adding one of the sticky classes to a grid element. The classes available are as follows.

token
  • "sticky":Provide sticky positioning across all viewport sizes.
  • "sticky-xs":Provide sticky positioning on Extra Small viewport and larger.
  • "sticky-s":Provide sticky positioning on Small viewport and larger.
  • "sticky-m":Provide sticky positioning on Medium viewport and larger.
  • "sticky-l":Provide sticky positioning on Large viewport and larger.
  • "sticky-xl":Provide sticky positioning on Extra Large viewport and larger.

In addition, the point at which the element becomes sticky can be adjusted using data-attributes. The default point is when the elment reaches the top of the viewport. By givng an integer value to one of the data-sticky-top attributes for an element, it will adjust the point at which the element becomes sticky. The value is the number of pixels from the top of the screen. The data attributes you can use are as follows.

  • "data-sticky-top":Defines stick point for all viewports.
  • "data-sticky-top-xs":Defines stick point for Extra Small viewport sizes.
  • "data-sticky-top-s":Defines stick point for Small viewport sizes.
  • "data-sticky-top-m":Defines stick point for Medium viewport sizes.
  • "data-sticky-top-l":Defines stick point for Large viewport sizes.
  • "data-sticky-top-xl":Defines stick point for Extra Large viewport sizes.
